## 设计决策(请 dream 时合并入 design-intent.md)

### 0.3 PORT_TCP_SERVER 的 send 语义 = broadcast

routes[].dst 配 tcp_server 端口名 = **广播给所有 accept 出来的 PORT_TCP_CLIENT,name 匹配为聚合键**。
理由:reactor.c:88-94 已经在 accept 时把每个 client 的 base.name 设为 server 的 base.name,broadcast 是这个设计的自然完成。
失败语义:任一 client 写失败 → WARN 日志 + 不主动 close(避免与 reactor 的 g_port_table 操作竞态,让 EPOLLIN read==0 路径走清理),继续给其他 client 发。
返回值:至少一个成功返回 len;全失败返回 -1;无 client 连接返回 0。

**遗留约束**:当前 fd 是阻塞 socket → 慢 client 会阻塞 reactor 一次写。PROJECT_CONTEXT v4 §14 不可阻塞契约的彻底解法(EAGAIN / 旁路 dispatcher 线程)放在阶段 1 的传输层重写里。本次 0.3 是占位的最小可工作语义,不解决阻塞。

代码位置:`port_manager.c:71-99`(`port_send_tcp_server_broadcast`),内联注释完整契约说明。

### 0.12 timeout = 100ms,signature 改返回 int

- **timeout 值 = 100ms**:比 watchdog 44s 兜底窗口短两个数量级,慢消费者下及时降级。常量 `QUEUE_PUSH_TIMEOUT_MS` 放在 `event_queue.h`,改值时 grep 一处即可。
- **drop 计数器**:`queue_get_drop_count()` 暴露,monotonic。U3 的 STATS 帧后续可直接调用。
- **签名改 `int`**:0=ok,-1=drop。caller(reactor.c / net_handler.c)当前不检查返回值不破坏既有行为(C 允许忽略),但留路给未来"我推不进去,我得做点事"场景。
- **WARN 日志格式**:`[queue] push timeout 100ms, drop dst=%s len=%d total_drops=%lu`,运维诊断字段齐。

## 新发现的契约/约束(候选 dream 入 design-intent)

1. **`net_handler.c` 是死代码且无法编译**:引用 `event_msg_t.type` 字段(已删)和 `EVT_NET_IN` 枚举(不存在)。Makefile 未引用,因此当前 build 干净。属于阶段 0 范围外,但建议主 agent 单独删除该文件 + `net_handler.h`,避免后续误编进来引发难解 bug。
2. **TCP_SERVER 的 broadcast 阻塞写隐患**:0.3 实现是阶段 0 占位,阻塞 socket + 慢 client 仍会让 reactor 单次 write 阻塞。这是 §14 契约在传输层的具体执行点,需要在阶段 1 协议层重写时改为非阻塞 + EAGAIN 处理。建议 dream 时把这条加到 design-intent §5"已知风险"。
3. **paramiko 4.0 + Windows + Armbian sshd 的 sftp.put() bug**:`sftp.put()` ENOENT,但 `sftp.open(path,"wb").write()` 流式写工作。RPD §6.5 部署脚本起手时必须用流式写。
