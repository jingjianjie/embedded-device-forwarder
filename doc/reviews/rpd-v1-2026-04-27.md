## 3. 新发现的隐性契约(冗余自 inbox,便于一站式阅读)

- **C-A — reactor 线程不可阻塞契约**
  reactor 单线程消费 + `port_send` 在慢 TCP 客户端下可能阻塞 + `event_queue.c:24-27` 的双 cond 阻塞 push,组合起来会**反向死锁整个 epoll 循环**。`5246de2` 顺手解决"队列满覆盖"的同时,把问题转移到了"reactor 阻塞"切面。
  **关键后果**: 破坏 R3 决议的兜底前提 — "reactor 还在跑就还在喂狗",hw watchdog 永不复位,整板假死。
  建议作为 design-intent §10 新条目入库:**reactor 线程不可被任何 send 路径阻塞**,否则 hw watchdog 兜底失效。

- **C-B — AF_UNIX SOCK_SEQPACKET 与 splice/sendfile 互斥**
  splice 只能在 pipe 一端工作,SEQPACKET socket 不可直接对接 splice/sendfile 零拷贝路径。这是协议传输层选型(RPD §4 决策表)与 SDK 零拷贝路径(§4.1 `ezr_send_zerocopy`)之间的**自相矛盾硬冲突**,必须在阶段 1 起手前解决。

- **C-C — 帧头 `seq` 字段语义未定义**
  R5 决议"激进无兼容字段、不为前向兼容预留 padding"原则下,`seq` 这种**协议骨架字段**最该现在就把语义定准:谁分配?是否查重?是否参与 ACK 配对?幂等键还是消息序号?
  **关键判断**: 后期改 `seq` 语义比给协议加字段还贵,因为修改语义会同时影响发送方、接收方、所有中间节点的解释逻辑。

## 4. 候选 open-questions(等 dream 时审议)

- **U3** — R-1 解法选型:
  - (a) reactor 旁路喂狗(独立喂狗线程,reactor 阻塞时 watchdog 仍超时)
  - (b) `queue_push` 带 timeout drop(超过 N ms 直接丢包 + WARN)
  - (c) 独立 dispatcher 线程只负责 send(reactor 永远只 epoll)
  代价不同(线程数、复杂度、丢包语义),需主 agent / 人类拍板。

- **U4** — R-2 解法选型:
  - (a) 改 SOCK_STREAM + 自管帧(放弃 SEQPACKET 边界保证)
  - (b) 删 zerocopy 承诺(SDK 退化为 memcpy)
  - (c) 上 SCM_RIGHTS + 共享内存旁路(zerocopy 仅大数据走旁路)
  影响 SDK 公开 API,选型会进入 ABI。

- **U5** — R-5 ez_router 自升级回滚机制选型:
  - (a) A/B slot(bootloader 切换,需双倍存储)
  - (b) 文件备份 + systemd Restart 计数(简单但回滚执行体不明)
  影响阶段 5 验收设计与硬件存储规划。
