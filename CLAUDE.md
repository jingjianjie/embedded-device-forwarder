# CLAUDE.md — 项目级 Agent 协作守则

适用于 **embedded-device-router**(`ez_router`)项目内的所有 Claude Code agent(主 agent + 任何 subagent)。

---

## 1. 基本对话行为
1. 如果有令你困惑或感到惊讶的事情,**立即停下并寻求确认**。
2. 先**复述需求理解**,再进行计划。
3. 规划 plan 时应当将人类提出的**原始需求保留为同目录下的 .md 并在 plan 中引用**。
4. **plan 执行后应确认原始要求是否满足**。
5. 假定用户不清楚代码实现。讲解 bug 时,重点是交代代码里已有的概念逻辑,再说明是如何引发的;背景应当清晰到足以**用一句话讲清楚 bug 的根因**。
6. 介绍逻辑时,可以用 **ASCII 概念图**进行介绍。
7. 每次回复以 `<人类待办>` 结尾(见 §9)。

---

## 2. 长探索复用 —— `project-structure-guide.md`

复用每次长探索,记录为 map:

- **2.1** 当定位修改目标或理解模块职责时需要**跨 3+ 文件追踪调用链**,或发现**文件/目录名与实际责任不符**,或发现**隐式依赖**(改 A 会坏 B)——任务完成后向 `project-structure-guide.md` 追加一条标准条目,覆盖:**核心发现 / 涉及文件 / 隐式依赖 / 推荐导航路径**。
- **2.2** 若相关条目已存在但与现实不符,**就地更新**。
- **2.3** 条目保持**架构/结构层级**,避免描述具体代码改动,以免随每次提交失效。
- **2.4** 如果是你落地的方案,**尽量避免引起跨文件探索**(与 §5.3 职责正交目标一致)。

---

## 3. 提交策略
- 主 agent 对粒度清晰的修改应**主动提交**。
- subagent 约束见 §10。

---

## 4. `PROJECT_CONTEXT.MD` 维护
1. 将项目进展记录到 `PROJECT_CONTEXT.MD`,或对其进行修改,尽量形成对项目**正交的描述**。
2. 侧重记录**改动对应的意图/当时的背景**,而非很容易就能查到并且容易过时的具体情况。
3. 文件开头备注**版本号**,每次修改 `PROJECT_CONTEXT` 时自增;条目要有对应的版本号,记录 lastmodified version。形如:
   ```
   86.[LM_194] 发生 xxx 时一定要先 xxx...
   ```
4. 如需要修改分发或者部署,**永远先改动本项目,再同步到分发包**。

---

## 5. 编码原则
1. **Clean 程度和性能优先**。
2. **改动不应该比问题本身更复杂**,必要时提出重构请求。
3. 代码应避免**状态或逻辑膨胀**。CLASS 和 DOMAIN 应足够正交。
4. **主动复用项目既有的工程措施** —— 改动前先扫一眼项目里 *别人遇到同类问题是怎么处理的*(utility / helper / pattern / 命名习惯 / 异常处理风格),**默认沿用**既有方式,而不是凭通用最佳实践另起炉灶。发现既有措施与你的方案冲突时,**优先怀疑自己方案没对齐**。
5. **YAGNI / 反过度抽象** —— 不为**未发生的扩展**提前抽象。在需求不明或扩展点未出现时,朴素实现优先;需要时再重构,而不是预留各种 hook / base class / strategy。
6. **新增函数和枚举必须有注释**。

---

## 6. 问题解决
1. **先调查清楚,再解决问题**。
2. 识别**隐性契约**:你的实现如果对调用方/用户有要求或者强假设,或者跨文件、跨抽象层耦合,必须记录到 `PROJECT_CONTEXT.MD`。
3. **隐性契约**应涵盖 "代码实现" 中反映的隐性需求。实现方式千万条,**如果实现方式令人惊讶,那么说明是为了避免什么坑**,应当作为契约记录。
4. **Chesterton's Fence** —— 看似多余或奇怪的代码**不要顺手删**。先假设它在避坑(参考 §6.2 / §6.3 隐性契约)。不理解成因就保留,或在 `<人类待办>` 里标注求证。注释/格式一类改动不算"顺手删",但应只做一件事。
5. **TEST AS DOC**:隐性契约优先用最小 **Test Case** 固化。若暂时无法自动化测试,允许使用最小复现样例代替,并在 `PROJECT_CONTEXT.MD` 中标注原因与适用范围。
6. **(重要) 解决问题前先快速分析逻辑切面**,避免落地代码侵入其他切面或者被侵入。建议思考:
   - 当前问题涉及哪些切面?
   - 本次只允许改哪个切面?
   - 哪些文件属于该切面?
   - 哪些外部行为不能改变?

**术语**:`逻辑切面` 指业务或功能视角下,分散在多个函数中的语义关联,这种跨代码的整体视角在局部修改时容易被忽略。

---

## 7. 测试习惯与 case 切面
- 默默记录**测试惯用的主要指令组合**。
- 考虑整个 **case 逻辑的切面**(case-level cross-cut)。

---

## 8. Agent 友好性
业务设计时,应当原生 **agent-friendly**。例如增加明确的信息返回,以方便 MCP-server 制作。

---

## 9. `<人类待办>` 机制
1. 人类会在对话中提出对自己的 prompt。通常形式:*"帮我增加暗场项目。提醒我要求你使用 pythonnet 验证拟合优度"*。因此你要在每次对话末尾**重复全部未完成的待办**,避免人类忘记。例如:
   ```
   <人类待办>: 1. 使用 pythonnet 验证拟合优度
   ```
2. 你可以在 `<人类待办>` 中要求人类**验证你 coding 时预设的重要假设**。保持简洁。

---

## 10. For subagent
1. 不要改写 `PROJECT_CONTEXT.MD` 和 `TEST`。
2. 谨慎提交。
